# FoodMap 前后端联调文档规范

## 1. 目的

`docs/integration/` 用于保存每次前后端联调的说明、验收标准和问题记录。

每次进入前后端联调前，主代理必须新建一个独立文件夹，避免联调目标、接口契约、测试数据、日志摘要和 BUG 复测结论散落在聊天记录或临时代码注释中。

## 2. 目录命名

推荐格式：

```text
docs/integration/<iteration>-<feature>/
```

示例：

```text
docs/integration/B1-auth-ios-backend/
docs/integration/B2-store-map-query/
docs/integration/B3-recommendation-create/
```

## 3. 必备文件

每个联调文件夹至少包含两个文件：

```text
integration-plan.md
issue-log.md
```

- `integration-plan.md`：记录本次联调功能、前后端职责、接口契约、验收标准、测试数据和提交记录。
- `issue-log.md`：记录联调失败、BUG、复现步骤、测试数据、前后端日志摘要、责任归因和复测结果。

可选附件目录：

```text
screenshots/
recordings/
network/
logs/
tests/
```

附件目录用于保存截图、录屏、脱敏网络摘要、脱敏日志摘要和测试结果。附件不是每次联调的强制文件，但如果联调结论依赖某类证据，应优先沉淀到对应目录。

## 4. 多 Agent 分工

联调阶段默认启用主代理 + 子代理协作：

- 主代理：负责流程推进、文件夹创建、联调说明维护、问题分派、最终验收结论。
- 后端观察子代理：负责检查后端服务、接口、入参、响应、数据库、日志和错误码。
- 前端观察子代理：负责检查 iOS 页面、交互、网络请求、状态展示、截图或录屏证据。
- 联调验收子代理：负责按 `integration-plan.md` 判断是否通过，按 `issue-log.md` 跟踪 BUG 复测。

子代理默认只读观察和反馈。需要修改代码时，必须由主代理按 `harness/agent-task-template.md` 重新明确写入范围。

## 5. 使用流程

1. 主代理在联调前创建本次联调文件夹。
2. 主代理复制或生成 `integration-plan.md` 和 `issue-log.md`。
3. 主代理在 `integration-plan.md` 中写清楚本次联调功能、迭代归属、联调安全点、前后端职责、接口契约、验收标准、环境和测试数据。
4. 主代理先判断是否达到联调准入点；如果前端或后端尚未到达节点，必须说明还需要补齐的最小开发范围。
5. 前端、后端代码到达联调安全点后，按 `integration-plan.md` 执行联调验收。
6. 联调失败时，在 `issue-log.md` 新增 BUG 条目，并补充复现步骤、测试数据、日志摘要和初步归因。
7. BUG 修复后必须记录复测结果。
8. 联调通过后，主代理在 `integration-plan.md` 写入最终结论、验证命令和提交记录。

## 6. 联调安全点

联调安全点表示前端和后端都到达可观测、可验证、可复测的节点。

准入检查至少包括：

- API 契约稳定：路径、方法、入参、响应、错误码明确。
- 后端可调用：接口能通过 curl、Postman 或自动测试调通。
- 前端可触发：页面或 ViewModel 能发起真实请求。
- 环境可运行：必要服务、数据库、中间件和配置可启动。
- 测试数据可准备：账号、门店、推荐、关系或媒体数据有创建和清理方式。
- 日志可追踪：能获得 `requestId`、`traceId`，失败时能查前后端日志。
- 范围已冻结：本次联调验什么、不验什么写入 `integration-plan.md`。

联调等级：

- `L0 契约联调`：只确认 API、DTO、字段、枚举和错误码。
- `L1 Mock 联调`：前端使用 mock 或后端占位响应验证页面流程。
- `L2 本地真实联调`：前端调用本地真实后端服务。
- `L3 环境联调`：在 OrbStack、测试环境或 prod-like 环境跑完整服务链路。
- `L4 验收联调`：包含日志证据、测试数据、BUG 复测和最终结论。

正式业务联调原则上至少从 `L2` 开始。权限、隐私、Token、可见范围、PUBLIC 统计、上传和支付类高风险能力必须推进到 `L3` 或 `L4` 才能认为完成。

## 7. 通过标准

一次联调结论只能使用以下四类：

- `通过`：必测场景全部执行并有证据，阻塞级和高风险 BUG 已关闭。
- `有条件通过`：主流程可用，仅存在明确记录的非阻塞问题，并有责任人和复测计划。
- `不通过`：主流程失败，或出现权限、隐私、Token、可见范围、PUBLIC 统计口径等高风险问题。
- `无法判定`：缺少环境、构建、测试、接口调用、截图/录屏、日志等关键证据。

一次联调只有在以下条件满足时才可以标记为 `通过`：

- `integration-plan.md` 中列出的必测场景全部通过。
- 前端页面、交互、异常态和网络请求符合约定。
- 后端接口、入参、响应、错误码和日志符合约定。
- `issue-log.md` 中本次阻塞级和高风险 BUG 已修复并复测通过。
- 无未解释的权限、隐私、Token、可见范围或 PUBLIC 统计口径风险。
- 已记录执行过的测试、构建、脚本或无法执行的原因。
