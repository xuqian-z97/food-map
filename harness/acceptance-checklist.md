# 迭代验收清单

每次重要迭代结束前，主代理必须按本清单验收。

## 1. 文档一致性

- [ ] 产品行为变化已同步 `CODEX-product.md`。
- [ ] 前端页面、导航、SDK、验收标准变化已同步 `CODEX-front.md`。
- [ ] 后端服务、数据库、API、事件、基础设施变化已同步 `CODEX-after.md`。
- [ ] 迭代顺序、项目结构、GitHub 同步规则变化已同步 `CODEX-gen.md`。
- [ ] 代理职责、文件边界、验收标准变化已同步 `AGENTS.md` 和 `.agents/*.md`。
- [ ] 项目 skills 变化已同步 `skills/*/SKILL.md`。
- [ ] harness 规则变化已同步 `harness/`。

## 2. 产品规则

- [ ] 菜名必填且为纯文字。
- [ ] 推荐图片、标签、推荐理由为可选。
- [ ] 用户可以选择可见范围。
- [ ] 好友/情侣只能查看被授权内容。
- [ ] 只有 `PUBLIC / 全部公开` 推荐进入全站统计。

## 3. 前端验收

- [ ] 登录后进入地图页。
- [ ] 地图页可以展示门店点位。
- [ ] 添加推荐流程轻量清晰。
- [ ] 可见范围选择不会误导用户。
- [ ] 页面包含加载、空、错误、无权限、网络不可用状态。
- [ ] Token 存储在 Keychain。
- [ ] 私密内容不输出到日志。

## 4. 后端验收

- [ ] 服务边界符合 `CODEX-after.md`。
- [ ] 每个服务使用独立数据库。
- [ ] 不存在跨服务直接查表。
- [ ] 数据库持久化实体、DTO、VO 分层清晰，Entity 不直接作为 API 契约。
- [ ] 固定字段由 `BaseEntity` 承载，业务主键保留在具体持久化实体中。
- [ ] 数据库访问使用 MyBatis Mapper + Mapper.xml。
- [ ] 标准单表 SQL 放在 `{EntityName}Mapper.java` / `{EntityName}Mapper.xml`。
- [ ] 复杂业务 SQL 放在 `{EntityName}DefineMapper.java` / `{EntityName}DefineMapper.xml`。
- [ ] application 层依赖仓储端口接口，基础设施实现没有向上穿透。
- [ ] 内存仓储仅用于测试或本地替身，不作为生产 profile 默认持久化实现。
- [ ] 写接口从 Token 获取当前用户身份。
- [ ] 推荐可见范围由后端校验。
- [ ] 社区统计只统计 `PUBLIC` 且 `NORMAL` 的推荐。
- [ ] 重复基础校验、值判断、脱敏、时间处理逻辑已沉淀到明确边界的 common 工具类。
- [ ] 未新增 `CommonUtils`、`StringUtils`、`DateUtils` 等无明确边界的万能工具类。
- [ ] 公共类、接口、枚举、异常、事件、配置类和中间件封装类具备类级 Javadoc 注释。
- [ ] 常量类和枚举类中的每一个常量、每一个枚举项都具备 Javadoc 注释。
- [ ] 公共方法、静态工厂方法、复杂业务判断、脱敏规则、幂等规则、权限规则和状态流转具备方法级注释。
- [ ] Redis、MQ、对象存储、内部服务调用通过统一封装或服务内基础设施适配器访问。
- [ ] Redis Key 遵守 `foodmap:{service}:{biz}:{version}:{key}` 格式，业务缓存设置 TTL。
- [ ] 业务事件发布通过统一事件发布接口，消费者具备幂等处理。
- [ ] 业务日志使用通用日志方法或统一日志封装，不随意字符串拼接关键日志。
- [ ] 日志不输出 Token、密码、密钥、完整手机号、完整邮箱、私密推荐内容或评论正文。
- [ ] 结构化日志包含 `requestId`、`traceId`、`serviceName`，能通过流水号串联一次接口调用的业务日志、SQL 日志和异常日志。
- [ ] 新增业务接口具备接口访问日志留痕，接口访问摘要符合独立日志 PostgreSQL 保留 15 天的设计。
- [ ] 新增关键业务动作已评估并补充审计日志，审计日志不保存敏感正文。
- [ ] SQL 日志按 `DEBUG` 语义设计，生产环境全量 SQL DEBUG 受动态配置控制，慢 SQL 和异常 SQL 具备 `WARN` 摘要。
- [ ] 涉及日志链路的变更已考虑 Kafka、Elasticsearch 7 天热查询和 OSS 冷归档策略。
- [ ] 列表接口支持分页。
- [ ] 地图接口支持边界框查询。

## 5. API 契约验收

- [ ] API 路径、方法、请求体、响应体清晰。
- [ ] 前端模型与后端 DTO 字段一致。
- [ ] 可见范围枚举一致。
- [ ] 错误结构清晰。
- [ ] 分页结构清晰。
- [ ] 地图 bbox 查询结构清晰。

## 6. 测试和构建

- [ ] 已执行可用的后端构建或测试。
- [ ] 已执行可用的 iOS 构建或测试。
- [ ] 已执行 harness 脚本。
- [ ] 未执行的测试已说明原因。

## 7. GitHub 同步

- [ ] 工作区状态已检查。
- [ ] 已提交重要迭代。
- [ ] 已推送到 `origin/main`。
- [ ] 本地分支与远程同步。

## 8. 部署验收

- [ ] 部署方案已同步 `deploy/`。
- [ ] ECS2 作为主应用服务器。
- [ ] ECS1 作为辅助服务器。
- [ ] 跨云服务器不通过裸公网访问 Redis、PostgreSQL、Nacos、RabbitMQ。
- [ ] 安全组只开放必要公网端口。
- [ ] OSS 凭证不进入 Git。
