# CODEX 产品文档

## 1. 文档目的

本文档用于定义 FoodMap App 的产品方向、功能范围、用户流程、权限规则和迭代边界。

后续所有产品、前端、后端变更都必须与本文档保持一致。如果产品方向发生变化，必须在修改代码之前或同时更新本文档。

## 2. 产品名称

暂定英文名：FoodMap

暂定中文名：

- 饭友地图
- 私房美食地图
- 好友美食地图

最终名称暂未确定。

## 3. 产品定位

FoodMap 是一款 iOS 优先的美食推荐地图 App，地图数据基于高德地图。

用户可以记录自己推荐的门店，在地图上标定位置，填写推荐菜单，上传可选图片，添加标签，并选择推荐内容的可见范围。

产品从“个人美食记录工具”开始，逐步发展为“好友/情侣/群组共享工具”，最终演进为“公开美食推荐社区”。

## 4. 核心价值

传统美食平台通常解决：

- 哪家店热门？
- 哪家店评分高？

FoodMap 重点解决：

- 我信任的人推荐了哪家店？
- 这家店具体应该点什么？
- 哪些菜值得吃，哪些菜需要避雷？
- 我的推荐内容可以只给自己、好友、情侣、群组或所有人看吗？

产品的核心差异不是单纯的地图，而是“门店下的推荐菜单”。

## 5. 产品原则

1. 先满足用户自己的记录需求。
2. 先做好熟人之间的可信推荐，再发展公开社区。
3. 权限和隐私边界必须清晰。
4. 只有“全部公开”的推荐内容才进入全站公开统计。
5. 地图是主入口。
6. 推荐菜单是最重要的内容对象。
7. 第一版即使没有大规模社区，也应该对个人用户有价值。

## 6. 目标用户

### 6.1 个人用户

希望记录自己美食地图的用户：

- 喜欢的门店
- 推荐的菜品
- 想复吃的店
- 私人备注和标签

### 6.2 好友用户

希望与可信好友互相分享美食推荐的用户。

### 6.3 情侣用户

希望拥有共同美食地图的情侣用户：

- 我推荐的店
- 对方推荐的店
- 两个人都喜欢的店
- 一起想去尝试的店

### 6.4 群组用户

希望在小圈子里共建美食地图的用户：

- 朋友群
- 公司午餐群
- 城市探店群
- 旅行美食群

### 6.5 社区用户

希望查看公开推荐的用户：

- 附近公开推荐
- 热门门店
- 热门菜品
- 按城市或标签浏览美食地图

## 7. 用户角色

| 角色 | 说明 | MVP 是否支持 |
| --- | --- | --- |
| 游客 | 未登录用户，后续可开放有限公开内容浏览 | 否 |
| 注册用户 | 可创建记录、推荐、关系和可见规则 | 是 |
| 好友 | 双方确认后的用户关系 | 是 |
| 情侣 | 双方确认后的特殊关系 | 是 |
| 群组成员 | 加入某个共享推荐群组的用户 | 暂缓 |
| 管理员 | 审核举报、管理公开内容、合并重复门店 | 暂缓 |

## 8. 核心对象

### 8.1 用户

用户是内容和关系的拥有者。

主要字段：

- 用户 ID
- 登录账号
- 昵称
- 头像
- 所在城市
- 简介
- 账号状态

### 8.2 门店

门店是地图上的餐厅或美食地点。

主要字段：

- 门店 ID
- 高德 POI ID
- 门店名称
- 地址
- 经度
- 纬度
- 城市编码
- 门店分类
- 创建用户 ID
- 门店状态

### 8.3 推荐菜单

推荐菜单是用户在某个门店下创建的推荐内容。

主要字段：

- 推荐 ID
- 门店 ID
- 用户 ID
- 菜名
- 推荐理由
- 价格
- 推荐程度
- 可见范围
- 状态
- 创建时间
- 更新时间

### 8.4 图片

图片为可选内容，可以附加在推荐菜单下。

菜单名称必须是纯文字，图片只作为补充信息。

### 8.5 评论

评论是用户围绕某条推荐菜单产生的互动内容。

MVP 阶段评论对象为推荐菜单，不直接对门店创建独立评论。门店详情页可以聚合展示该门店下推荐菜单的评论。

评论主要字段：

- 评论 ID
- 推荐 ID
- 评论用户 ID
- 评论用户昵称快照
- 评论内容
- 评论图片
- 父评论 ID
- 评论状态
- 创建时间
- 更新时间

评论规则：

- 用户只能查看自己有权限查看的推荐菜单下的评论。
- 用户只能评论自己有权限查看的推荐菜单。
- 评论人昵称需要冗余保存为快照，避免用户改名后历史评论展示不稳定。
- 每条评论最多支持 3 张图片。
- 评论图片为可选内容。
- 评论图片通过媒体服务保存元数据，推荐服务只保存媒体业务主键引用。
- 非 PUBLIC 推荐下的评论不能进入全站社区统计。

### 8.6 标签

标签用于描述口味、菜系、场景或特点。

示例：

- 辣
- 甜
- 清淡
- 约会
- 工作餐
- 咖啡
- 性价比高
- 隐藏小店

### 8.7 用户关系

用户关系用于控制社交可见范围：

- 好友关系
- 情侣关系
- 群组成员关系
- 拉黑关系

### 8.8 可见规则

可见规则决定推荐菜单可以被谁看到。

可见权限必须由后端强制校验，不能只依赖前端过滤。

## 9. 可见范围规则

| 可见范围 | 谁可以看 | 是否进入全站统计 |
| --- | --- | --- |
| 仅自己可见 | 只有作者 | 否 |
| 指定用户可见 | 作者和被指定用户 | 否 |
| 好友可见 | 作者和作者好友 | 否 |
| 情侣可见 | 作者和绑定情侣 | 否 |
| 群组可见 | 作者和指定群组成员 | 否 |
| 全部公开 | 所有用户 | 是 |

重要规则：

只有“全部公开”的推荐内容才进入全站公开统计。

指定用户可见和群组可见的内容，后续可以支持私有统计或群组内统计，但不能混入全站公开统计。

## 10. MVP 功能范围

### 10.1 MVP 包含功能

1. 账号注册和登录。
2. 用户资料。
3. 基于高德地图的地图展示。
4. 通过高德 POI 搜索门店。
5. 当 POI 不存在时支持手动创建门店。
6. 创建推荐菜单。
7. 可选上传推荐图片。
8. 推荐标签。
9. 推荐内容可见范围设置。
10. 我的美食地图。
11. 好友申请和好友关系。
12. 情侣绑定。
13. 查询好友/情侣可见推荐。
14. 推荐菜单评论。
15. 评论图片上传。
16. 全部公开内容的基础统计。
17. 附近公开推荐列表。
18. 当前城市高德离线底图下载和更新。
19. 离线或弱网时展示最近缓存的门店点位和门店摘要。

### 10.2 MVP 暂不包含功能

1. iCloud 同步。
2. 第三方登录。
3. 点赞。
4. 踩。
5. 打星评分。
6. 收藏。
7. 复杂信息流算法。
8. 城市公开榜单。
9. 群组共享。
10. 管理后台。
11. AI 菜单总结。
12. 好友、情侣、指定用户、群组推荐详情和评论内容的完整离线浏览。

### 10.3 离线地图和本地缓存范围

FoodMap MVP 支持高德离线底图，用于保证用户在无网或弱网时仍能看到基础地图。

离线底图规则：

- App 首次打开时优先请求定位权限，定位成功后识别当前城市并下载该城市高德离线底图。
- 用户拒绝定位、定位失败或定位城市无法匹配离线包时，引导用户手动选择城市。
- 用户在城市切换入口中切换城市后，保存新的当前城市并触发该城市离线底图下载。
- App 每次打开时尝试定位当前城市；定位城市与用户选择城市不一致时，提示用户是否切换。
- Wi-Fi 环境下自动静默检查并更新当前城市离线底图，不提供关闭入口。
- 蜂窝网络下载离线底图需要用户在设置中开启，默认关闭。
- 本地最多保留 3 个城市离线底图包；超过后按最近最少使用自动删除旧城市，当前城市不能被自动删除。

业务数据缓存规则：

- 可缓存当前城市最近加载过的门店点位摘要、门店摘要、用户自己的推荐摘要和 PUBLIC 公开内容摘要。
- 离线或接口超时时，地图页可展示最近缓存的门店点位。
- 用户可以点击缓存点位进入门店摘要页，页面顶部先展示“离线缓存”标识，后续根据真实体验再决定是否弱化。
- 离线状态下只能查看缓存摘要，不允许新增、编辑、评论或上传图片。
- 好友、情侣、指定用户、群组推荐详情、评论正文和评论图片不做离线展示，避免服务端权限变化后本地继续泄露内容。
- 退出登录、账号切换或登录状态失效时，必须清理当前账号的业务缓存。

## 11. 核心用户流程

### 11.1 注册和登录

1. 用户打开 App。
2. 用户选择手机号、邮箱或账号名登录。
3. 后端校验登录凭证。
4. App 安全存储访问令牌。
5. 用户进入地图首页。

### 11.2 添加门店和推荐菜单

1. 用户在地图页点击添加。
2. 用户输入门店关键词。
3. App 调用后端门店搜索接口。
4. 后端调用高德 POI 接口或返回本地已保存门店。
5. 用户选择门店。
6. 用户填写菜名。
7. 用户可选填写推荐理由、价格、推荐程度、标签和图片。
8. 用户选择可见范围。
9. 后端保存推荐内容。
10. 如果可见范围为全部公开，后端发送事件给社区统计服务。

### 11.3 查看我的地图

1. 用户打开地图。
2. App 根据当前地图视野请求可见门店和推荐内容。
3. 后端返回当前用户有权查看的数据。
4. App 渲染地图点位。
5. 用户点击点位查看门店摘要。
6. 用户进入门店详情查看推荐菜单。

### 11.4 添加好友

1. 用户通过账号名、手机号或邮箱搜索其他用户。
2. 用户发送好友申请。
3. 对方收到申请。
4. 对方接受后双方成为好友。
5. 双方可以查看设置为好友可见的推荐内容。

### 11.5 绑定情侣

1. 用户从好友中选择一个用户。
2. 用户发送情侣绑定申请。
3. 对方接受申请。
4. 情侣关系生效。
5. 双方可以查看设置为情侣可见的推荐内容。

### 11.6 公开社区统计

1. 用户创建或更新全部公开的推荐内容。
2. 推荐服务发送领域事件。
3. 社区服务消费事件。
4. 社区统计数据更新。
5. 公开地图和热门列表展示最新结果。

## 13. 非功能需求

### 13.1 隐私

- 后端必须强制校验所有可见权限。
- 私密推荐不能出现在公开接口中。
- 评论和评论图片必须继承所属推荐菜单的可见范围。
- 指定用户可见推荐不能进入公开统计。
- 群组可见推荐不能进入全站统计。

### 13.2 性能

- 地图查询必须支持按视野范围分页加载。
- 地图缩放层级较低时，后端应返回聚合点或简化点位。
- 公开热门列表应使用缓存。

### 13.3 可用性

- App 采用在线优先模式。
- 网络不可用时展示不可用提示，并允许用户查看高德离线底图。
- MVP 支持弱网或离线时查看最近缓存的门店点位和门店摘要。
- 离线状态下不允许新增、编辑、评论或上传图片。
- 好友、情侣、指定用户、群组推荐详情和评论内容不做完整离线浏览。

### 13.4 安全

- 密码必须加密哈希存储。
- Token 必须有过期时间。
- Refresh Token 必须支持撤销。
- API 必须校验资源归属和可见权限。
- 图片上传必须校验文件类型和大小。

### 13.5 内容审核

MVP 需要为公开内容预留状态字段：

- 正常
- 待审核
- 已隐藏
- 已删除

真正的后台审核系统可以后续建设。

## 14. 产品路线图

### 阶段一：MVP

目标：个人地图 + 好友/情侣共享 + 公开统计。

主要功能：

- 登录
- 地图
- 门店搜索/创建
- 推荐菜单创建
- 推荐菜单评论
- 评论图片
- 可见范围
- 好友关系
- 情侣关系
- 公开统计

### 阶段二：群组共享

目标：支持小圈子共建地图。

主要功能：

- 创建群组
- 邀请成员
- 群组可见推荐
- 群组地图
- 群组内统计

### 阶段三：社区化

目标：建设公开发现能力。

主要功能：

- 公开推荐流
- 附近热门门店
- 热门菜品
- 标签页
- 城市页
- 内容举报

### 阶段四：智能化和增长

目标：提升发现效率和用户留存。

主要功能：

- 收藏
- 通知
- 点赞、踩、打星评分
- AI 门店/菜品总结
- 个性化推荐
- 可分享公开美食地图
